## Current integrated flow

The active integrated package flow is now:

1. Phase 1: math optimisation by default via `dse_tool.agents.phase1_mathopt_agent.Phase1MathOptAgent` with CP-SAT selected by default

Key current findings:

- Under the default `cost_only` Phase 2 objective, the Pixhawk UAV model still places only:
  - `pep_telem1`
  - `ps_fmu`
- Under the optional `control_plane` resilience-aware objective, the revised dual-PS architecture is solver-discovered without forcing:
  - `pep_telem1`
  - `pep_can1`
  - `pep_can2`
  - `pep_px4io`
  - `ps_fmu`
  - `ps_io`
- That benefit is currently localized:
  - `ps_fmu` compromise no longer ungovers the actuator and PX4IO protection paths
  - but the top-level worst-case risk metric does not yet improve

Control-plane sensitivity result:

- behavior is identical across all three Phase 1 strategies in the sampled grid
- the revised dual-PS architecture selects split governance in `12/16` sampled weight pairs per strategy
- the observed threshold is stable in the sampled grid:
  - once the safety-critical PEP penalty reaches `250`, the revised architecture selects the dual-PS split for every sampled concentration weight
- the baseline architecture never reaches split governance
  - it either stays telemetry-only
  - or adds safety PEPs under the same single PS

## Remaining limitations

- Runtime remains tc9-specific by design.
- Legacy compatibility tolerance still exists for:
  - `ungovernerd_pep` parser support
  - `max_bufg` alias support
- The resilience-aware Phase 2 objective currently uses explicit proxy weights.
  These weights are now sensitivity-tested, but they are still modeling choices rather than empirically calibrated constants.
- End-of-suite CPU usage can still drop once the last long solver case becomes the only remaining active worker. The parallel runner reduces this effect by front-loading heavy tests, but it cannot eliminate the final long-tail case without further splitting that test.
