## Architecture

- **Core DSE flow**: Phases 1-2 construct a protected baseline. Phase 1 performs security/realtime-detection feature selection with CIA-style availability risk and DoS exposure; the package defaults to the ILP Phase 1 backend and falls back to ASP/Clingo when ILP is unavailable or UNSAT. Phase 2 performs ZTA policy synthesis; Phase 3 performs resilience assessment over that baseline using service continuity, mission-capability availability, blast radius, and capability-loss metrics.
- **Three strategies**: `max_security`, `min_resources`, `balanced` â€” shared package-level strategy names used by both the ILP default backend and the ASP fallback backend.
- ASP encodings live in `Clingo/` directory. Instance facts are generated by `dse_tool/core/asp_generator.py`.
- GUI is Tkinter-based: `main_window.py` (root), `network_editor.py` (canvas), `results_panel.py` (output), `progress_panel.py` (log).
- Agent layer: `orchestrator.py` coordinates `phase1_agent.py`, `phase2_agent.py`, `phase3_agent.py`, and optionally `runtime_agent.py`. In paper terminology, Phases 1-2 are upstream baseline construction and Phase 3 is the RASACC resilience assessment.
- Core data flow: `NetworkModel` â†’ `ASPGenerator.generate()` â†’ facts string. Phase 1 then uses `Phase1MathOptAgent` by default with the CP-SAT backend (CBC optional, `Phase1Agent` on ASP fallback); Phases 2-3 use `ClingoRunner.solve()` / `solve_scenario()` and `SolutionParser` to build `SolutionResult`.

## Key Data Structures

- `NetworkModel` (asp_generator.py): Complete SoC description â€” components, links, services, access needs, capabilities, ZTA topology, trust anchors, mission phases.
- `MissionCapability` (asp_generator.py): High-level mission function with required services, components, access paths, and criticality (essential/important/optional).
- `Component` (asp_generator.py): Hardware component with domain, impact scores, exploitability, trust properties.
- `ScenarioResult` (solution_parser.py): Phase 3 output per scenario â€” blast radii, risks, services, control plane, capability status.
- `SolutionResult` (solution_parser.py): Aggregates Phase 1 + Phase 2 + Phase 3 per strategy. Optional `runtime_results` and `joint_runtime` fields for runtime analysis.
- `RuntimeScenario` (solution_parser.py): Frozen dataclass defining a runtime monitoring scenario with observed signals.
- `RuntimeAdaptiveResult` (solution_parser.py): Output of a single adaptive runtime solve â€” monitors, modes, trust states, response actions.
- `JointPhase2RuntimeResult` (solution_parser.py): Output of joint Phase 2 + runtime synthesis. Has `to_phase2_result()` for downstream Phase 3 compatibility.

## Resilience Encoding Design

- `service_live_count` excludes both `failed(C)` and `compromised(C)` members â€” a compromised service member is treated as unavailable.
- `asset_available` excludes compromised owners; `asset_compromised/1` tracks compromised-owner assets separately.
- Attack paths (`attack_step`) use `effective_adjacent` (firewall-aware). Structural attack paths (`structural_attack_step`) use `active_adjacent` for worst-case analysis.
- `node_cut_off(C)` checks that no *master* can reach C, not just any node.
- `protection_discount` combines `security_discount` (prevention) + `realtime_discount` (realtime detection) + `audit_discount` (post-compromise visibility), capped at 10.
- `capability_access_unmediated` flags access paths where the PEP is bypassed â€” capability is degraded (not lost).
- `system_functional` is emitted for capability-free topologies, so Phase 3 does not default them to non-functional.
- `capability_degraded` bypass variant is guarded by `not capability_lost(Cap)` to prevent simultaneous lost+degraded.
- Resilience score = 40% blast radius + 40% capability retention + 20% control plane health (3-axis). When function-support data is present, switches to 4-axis: 35% blast radius + 30% capability retention + 20% control plane health + 15% function diversity.
- `FunctionSupport` (asp_generator.py): Cross-service functional contribution with modality metadata. Fields: function, component, modality, quality (0-100), bus. Used by the Python Phase 3 evaluator (`phase3_fast_agent.py`) for diversity-aware ok/degraded/lost assessment.
- **Known parity gap**: Python Phase 3 ignores trust-level amplification (`_trust_amp` returns 0) because Phase 2 does not serialize `p2_trust_level` facts. ASP Phase 3 uses ±0.3x amplification. Do not mix Python and ASP Phase 3 results in the same comparison until this is resolved.
- `SolutionRanker` accepts `max_luts`/`max_power` from `NetworkModel.system_caps`; module-level defaults remain as fallback when caps are absent.
- `solve_scenario()` respects the ClingoRunner timeout â€” no unbounded solve calls.

## Comparison / Report

- `comparison.py` is fully topology-agnostic â€” no hardcoded component names, PEP names, or LUT counts.
- `ComparisonEngine` and `generate_report_text()` accept `max_luts`, `max_power`, `max_ffs` parameters â€” resource percentages reflect the actual target platform.
- Orchestrator passes `system_caps` to the report generator automatically.
- Recommendations are auto-generated from actual Phase 1/2/3 data (trust gaps, excess privileges, capability risks).
- Report includes "MISSION CAPABILITY SUMMARY" section with functional/degraded/non-functional counts per strategy.
- Comparison table includes "Cap. Retention" and "Non-Func Scenarios" rows.

## Topology Validation

- `ASPGenerator.validate_topology()` checks for structural UNSAT risks before running the solver.
- Checks: critical-IP FW coverage for low-trust masters, FW governance, ip_loc consistency, safety-critical receiver typing.
- Orchestrator runs validation at startup and logs warnings via the progress queue.
- Phase 2 UNSAT diagnosis (`_diagnose_unsat`) tests three constraint relaxations independently (FW placement, safety isolation, FW governance).

## Runtime Adaptive Monitoring

- **TC9-only in this version.** Runtime LP files, scenario definitions, and diagnostic logic are tc9-specific. Non-tc9 topologies are rejected before solving.
- **Terminology boundary**: Phase 1 availability is CIA-A risk / DoS exposure in the protected baseline. Phase 3 availability is service continuity / mission-capability availability under scenarios. Runtime is a post-Phase-2 extension beyond the core RASACC workflow, not a redefinition of Phase 1 availability.
- **Two runtime modes** (distinct integration shapes):
  - **Adaptive runtime** (`solve_adaptive`): requires Phase1Result + Phase2Result. Evaluates monitoring scenarios against a decided control plane. Raises `RuntimeError` on UNSAT/timeout/error.
  - **Joint runtime** (`solve_joint`): requires Phase1Result only. Replaces Phase 2 with co-optimized FW/PS/monitor placement. Never raises â€” returns `satisfiable=False` on failure.
- **Orchestrator flags**: `run_adaptive_runtime=True` (after Phase 2), `run_joint_runtime=True` (replaces Phase 2).
- **CLI runner**: `python run_runtime.py` (standard) or `python run_runtime.py --joint` (joint mode).
- **`joint.to_phase2_result()`** reuses `SolutionParser.parse_phase2()` on joint atoms for full Phase 2 field parity.
- **`joint.as_runtime_facts()`** emits `deployed_pep`, `deployed_ps`, `deployed_monitor`, and `p2_allow` facts needed by the adaptive encoding.
- **5 tc9 scenarios**: baseline, dma_rate_spike, dma_privilege_creep, c8_sequence_anomaly, ps0_policy_tamper.
- Runtime LP files: `runtime_adaptive_tc9_enc.lp`, `runtime_monitor_tc9_inst.lp`, `zta_policy_runtime_enc.lp`.

## Known Issues / Gotchas

- Legacy standalone artifacts are quarantined under `legacy/` and are not part of the integrated package path. `_LEGACY_LP_NAMES` is empty â€” the generalized `resilience_enc.lp` is the only Phase 3 encoding used by the orchestrator.
- `runtime_monitor_tc9_inst.lp`, `runtime_adaptive_tc9_enc.lp`, `tgt_system_tc9_inst.lp` still have tc9 in their names (not yet used by the GUI pipeline).
- `ScenarioResult.system_functional` still defaults to `False` in Python, but the generalized resilience encoding now emits `system_functional` for capability-free topologies.
- `ScenarioResult.active_ps_count` defaults to `0` (not 2) â€” reflects actual solver output.
- `init_enc.lp` selects features per `asset(C,_,_)` tuple, not per component. Multiple asset-operations on the same component create independent choice atoms, but Clingo's set semantics and the optimization objective cause the solver to assign the same feature. The parser's regex filter retains only component-level entries.
- `attack_step` depth bound is configurable via `system_capability(max_attack_depth, N)`; when absent it defaults to 5 hops. Intermediate node revisits are possible on cyclic topologies; `min_attack_distance` extracts the shortest path so metrics are correct.
- `scenario_asset_risk` uses `has_action_risk/0` guard to select between 3-arg (per-action) and 2-arg (per-asset) p1_risk exclusively â€” no circular dependency.
- Scenario generation validates component names against ASP-emittable set; scenarios with invalid names are silently skipped.
